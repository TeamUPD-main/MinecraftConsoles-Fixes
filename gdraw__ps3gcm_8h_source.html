20IDOC extern int gdraw_GCM_SetResourceMemory(gdraw_gcm_resourcetype type, S32 num_handles, void *ptr, S32 num_bytes);
21/* This sets up the resource pools that GDraw uses for its video memory management.
22
23 It sets both the number of handles and the address and size of memory to use.
24 GDraw keeps track of allocations in each pool, and will free old resources in
25 a LRU manner to make space if one of the limits is about to be exceeded. It will
26 also automatically defragment memory if necessary to fulfill an allocation
27 request. The pointer should be either to RSX local memory or to RSX-mapped
28 main memory. It also needs to be aligned to a cache line boundary (128 bytes).
29
30 You need to set up all of the resource pools before you can start rendering.
31 If you modify this at runtime, you need to call IggyPlayerFlushAll on all
32 active Iggys (if any) since this call invalidates all resource handles they
33 currently hold.
34
35 If any Iggy draw calls are in flight, this call will block waiting for
36 those calls to finish (i.e. for the resource memory to become
37 unused).
38*/

41IDOC extern int gdraw_GCM_SetRendertargetMemory(void *ptr, S32 num_bytes, S32 width, S32 height, S32 pitch);
42/* Specifies the start address, size and physical dimensions of the region used to allocate render targets from.
43
44 All render targets allocated by GDraw have the same size (width x height pixels) and will use the same pitch;
45 thus you can pack them all into one tile region, which is recommended from a performance perspective.
46 Pitch must allow for 4-byte-per-pixel rendertargets (so pitch must be at least width*4).
47
48 On other platforms, Iggy does not require you to correctly specify the width and height of the
49 rendertargets; the size you specify is a hint, and GDraw will detect the actual
50 needed size and reallocate the rendertargets to be large enough as needed.
51 Because of the need to set pitch accurately, this is not true on the PS3; instead,
52 the width and height you specify here _must_ be sufficiently large. To be explicit,
53 each of width and height be as large as the largest Iggy you will render in that
54 dimension, or if you're rendering with tiles, the largest tile you'll render,
55 including padding (possibly on both sides).
56
57 For example, if all you ever render is a 400x300 Iggy, you could call this
58 with width,height=300. If all you render is a 1920x1080 Iggy split into two
59 horizontal tiles with 8 pixels of padding, you could call this with
60 width=960+8*2, height=1080+8*2. (This is assuming there is padding on all
61 sides of the tile; in practice, with this particular arrangement, there is
62 only ever one column of horizontal padding, so technically it is large
63 enough to use <960+8,1080>. I prefer to just always pad all sides so that
64 I don't have to worry about how things change depending on 3-tiles-in-a-row
65 versus 4-quadrant-tiles, but this does waste some memory.)
66
67 If you render multiple Iggys in one frame (or, generally, during a single "major mode",
68 since you don't want to reallocate rendertargets on the fly), you must take the max
69 of each dimension. For example, if you were to render:
70 $* A 500x200 Iggy
71 $* An 800x100 Iggy
72 $* One Iggy using two 400x300 tiles, each with 8 pixels of padding
73 Then the necessary width would be max(500,800,300+8*2) and the necessary height
74 would be max(200,100,400+8*2); in other words, you would need to call this
75 function with width and height no smaller than <800,416>.
76
77 You can use this memory for your own purposes as well. IggyPlayerDraw* calls will write to it,
78 and you should not touch it _during_ IggyDraw calls. GDraw only touches this memory from
79 the GPU, so you must only access it from the GPU.
80
81 Calling this function causes GDraw to wait for all outstanding GDraw operations
82 to drain from the GPU pipeline.
83*/

99/* Creates a GDraw context for rendering using GCM. You need to pass in the GcmContextData
