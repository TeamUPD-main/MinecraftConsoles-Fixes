38IDOC extern int gdraw_orbis_SetResourceMemory(gdraw_orbis_resourcetype type, S32 num_handles, void *ptr, S32 num_bytes);
39/* Sets up the resource pools that GDraw uses for its video memory management.
40
41 It sets both the number of handles and the address and size of memory to use.
42 GDraw keeps track of allocations in each pool, and will free old resources in
43 a LRU manner to make space if one of the limits is about to be exceeded. It will
44 also automatically defragment memory if necessary to fulfill an allocation
45 request.
46
47 You need to set up all of the resource pools before you can start rendering.
48 If you modify this at runtime, you need to call IggyPlayerFlushAll on all
49 active Iggys (if any) since this call invalidates all resource handles they
50 currently hold.
51
52 SetResourceMemory takes a void* argument for the address of the resource pool.
53 Pass in nullptr and zero bytes to reset a specific pool.
54
55 Resource pool memory has certain alignment requirements - see the #defines
56 above. If you pass in an unaligned pointer, GDraw will automatically clip off
57 some bytes at the front to make the buffer aligned - in other words, you get
58 somewhat less usable bytes, but it should work fine.
59
60 "rendertarget" memory is only used during Iggy rendering. In other words, you
61 are free to use that memory for other purposes such as your own render targets
62 or depth buffers as long as it doesn't need to be preserved across IggyPlayerDraw*
63 calls.
64
65 If any Iggy draw calls are in flight, this call will block waiting for
66 those calls to finish (i.e. for the resource memory to become
67 unused).
68*/

99IDOC extern void gdraw_orbis_Begin(sce::Gnmx::GfxContext *context, void *staging_buffer, U32 staging_buf_bytes);
100/* This sets the GfxContext that GDraw writes its commands to. It also sets the
101 address and size of the staging buffer used for dynamic vertex data and in-frame
102 resource uploads. Any GDraw / Iggy rendering calls outside a Begin / End bracket
103 are an error and will be treated as such. The GfxContext must be live during the
104 entire Begin / End bracket.
105
106 GDraw maintains a persistent resource cache shared across all Iggys. Because of this,
107 it is *vital* that all GfxContexts generated from GDraw be kicked in the order
108 they were generated, or the resource pools might get corrupted. Hence it's recommended
109 to use only one GfxContext for all Iggy rendering during a frame.
110
111 The staging buffer should be allocated in write-combined memory. If it is of insufficient
112 size, GDraw will not be able to allocate dynamic vertex data or upload new texture/vertex
113 buffer data during some frames, resulting in glitching! (When this happens, it will be
114 reported as a warning, so make sure to install a warning callback).
115
116 The user is expected to handle GfxContext and staging buffer synchronization; there are
117 no safeguards on the GDraw side to prevent CPU-GPU race conditions. Please make sure that
118 the GPU is finished with a GfxContext / staging buffer before reusing its memory!
119 Typically, this would be accomplished by double buffering everything. */
120
121IDOC extern void gdraw_orbis_End(gdraw_orbis_staging_stats *staging_stats);
122/* This marks the end of GDraw rendering for a frame. It also triggers end-of-frame processing,
123 which is important for GDraw's internal resource management. GDraw will not touch the GfxContext
124 or staging buffer after this call, so you are free to append other rendering commands after
125 this call returns.
126
127 staging_stats will be filled with stats for the staging buffer, denoting how much memory
128 was actually used and which allocations were attempted. If you're not interested, just
129 pass nullptr. */
