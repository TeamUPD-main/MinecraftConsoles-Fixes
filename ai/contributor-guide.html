Contributor Guide

This guide provides practical information for new and existing contributors to the Minecraft Consoles Fixes project. Understanding the project's structure and common code patterns will help you make effective contributions.

Project Structure Overview

As detailed in the Architecture Overview, the project is organized into distinct layers:

  • Minecraft.World: Core game logic (e.g., physics, entities, items, blocks).
  • Minecraft.Client: Client-side application (e.g., UI, rendering, input).
  • Minecraft.Server: Server-side application (e.g., multiplayer, access control).
  • Common: Shared utilities and definitions.
  • 4JLibs, Durango, Orbis: Platform-specific code and abstractions for Xbox and PlayStation.

Most contributions will involve changes within one or more of these modules, often requiring an understanding of their interactions.

Key Areas for Contribution

Here are common areas where you might contribute and the relevant modules:

  • Modifying Gameplay Mechanics (Minecraft.World)

    If you're looking to change how entities behave, introduce new item functionalities, alter block properties, or adjust physics, your work will primarily be in the Minecraft.World module.

    • Physics: Modify AABB.h, Vec3.h for collision and movement logic.
    • Entities/Mobs/Players: Extend classes like LivingEntity.h, Mob.h, Player.h. Adjust player abilities through Abilities.h.
    • Items/Blocks: Work with Item.h, Tile.h, and their derived classes.
    • Status Effects: Implement new effects or modify existing ones using MobEffect.h and specific effects like AbsoptionMobEffect.h.

  • Extending Client Features (Minecraft.Client)

    Contributions related to the user interface, rendering, or client-side platform integration belong here.

    • User Interface: Implement new screens or GUI elements by extending Screen.h or GuiComponent.h. Existing container UIs use AbstractContainerScreen.h.
    • Rendering: Adjust how items are drawn via ItemRenderer.h or manage textures using Textures.h and AbstractTexturePack.h.
    • Input: Modify input handling or key mappings via KeyMapping.h.

  • Enhancing Server Management (Minecraft.Server)

    For multiplayer-focused changes, especially regarding server management and player access, the Minecraft.Server module is the place to look.

    • Access Control: Implement new ban or whitelist rules, or modify existing ones within Access.h, BanManager.h, and WhitelistManager.h.
    • Server Logic: Changes to the main server loop or game session management will be around ServerMain.cpp.

  • Platform-Specific Implementations (4JLibs, Durango, Orbis)

    If your contribution involves specific console features, such as user file management (UGC, saves) or integrating with platform achievements:

    • User File Management: Refer to 4J_xtms.h for functions like XSGX_XTmsPutUserFile and XSGX_XTmsCancelPutUserFile.
    • Achievements: Investigate AchievementManager.h in the platform-specific directories for integration with console achievement systems.

  • Core Utilities (Common)

    For general-purpose enhancements or bug fixes affecting definitions, game rules, telemetry, or audio, the Common module is relevant.

    • Game Rules: Modify existing or add new game rules in ConsoleGameRules.h.
    • Telemetry: Understand how game data is collected and reported by examining TelemetryManager.h.

Understanding the Codebase

  • Dependencies: Use Include Graphs

    Doxygen provides include dependency graphs. When working on a file, examine its incdepgraph (what it includes) and invincdepgraph (what includes it) to understand its place in the system and potential side effects of changes. This is crucial for navigating the large codebase.

  • Data Structures: NBT System

    The game extensively uses the NBT (Named Binary Tag) format for data serialization. You'll frequently encounter classes like CompoundTag, ListTag, ByteTag, IntTag, StringTag, FloatTag, etc. Understanding how these tags work and how to read/write them using NbtIo.h and stream classes is fundamental for handling persistent game data (e.g., player data, world chunks).

  • Input/Output Operations

    File and network I/O often uses stream abstractions like InputStream.h, OutputStream.h, DataInput.h, and DataOutput.h, with specialized implementations for files (FileInputStream, FileOutputStream) and compression (GZIPInputStream, GZIPOutputStream).

  • Concurrency and Thread Safety

    In server-side code, especially within the Access module, be aware of concurrency. Files like Access.cpp explicitly use std::mutex (e.g., stateLock, writeLock) to protect shared data. Ensure any new concurrent code adheres to proper synchronization mechanisms to prevent race conditions.

  • Platform Specifics

    Code within 4JLibs, Durango, or Orbis directories often interacts with console-specific APIs. Be cautious when modifying these, as changes might only apply to a particular platform or require specific SDK knowledge.

Practical Advice

  • Respect Existing Structure: Maintain the current directory structure and file organization to ensure consistency.

  • Review Headers: Always review the included headers in a .cpp file (e.g., #include "stdafx.h" for precompiled headers) to understand local dependencies and the compilation environment.

  • Utilize Shared Utilities: Before writing new utility functions, check the Common module and various *Utils.h files (e.g., StringUtils.h, FileUtils.h, AccessStorageUtils.h) for existing implementations.

  • Callback Patterns: Be aware of callback patterns for asynchronous operations, such as TMSCLIENT_PROG_CALLBACK, which are common in platform integration code.

Start Small: For new contributors, begin with minor bug fixes or small feature enhancements to familiarize yourself with the codebase and workflow.

Do Not Invent Systems: Adhere to the existing architecture and system design. Do not introduce entirely new architectural patterns or major components without prior discussion and agreement.
