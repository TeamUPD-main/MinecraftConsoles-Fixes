58IDOC extern int gdraw_psp2_SetResourceMemory(gdraw_psp2_resourcetype type, S32 num_handles, void *ptr, S32 num_bytes);
59/* Sets up the resource pools that GDraw uses for its video memory management.
60
61 It sets both the number of handles and the address and size of memory to use.
62 GDraw keeps track of allocations in each pool, and will free old resources in
63 a LRU manner to make space if one of the limits is about to be exceeded. It will
64 also automatically defragment memory if necessary to fulfill an allocation
65 request.
66
67 "ptr" points to the address of the resource pool. This memory needs to be
68 mapped to the GPU and *writeable*. If it isn't, the GPU will crash during
69 either this function or CreateContext!
70
71 Pass in nullptr for "ptr" and zero "num_bytes" to free the memory allocated to
72 a specific pool.
73
74 GDraw can run into cases where resource memory gets fragmented; we defragment
75 automatically in that case. However, to make this work, GDraw on PSP2 needs
76 resource pool memory equivalent to *twice* the largest working set in any
77 scene. So for example, if you use 10MB worth of textures, GDraw needs at least
78 a 20MB texture pool! On other platforms, we can avoid this extra cost by
79 draining the GPU pipeline in the middle of a frame in certain rare cases, but
80 doing so on PSP2 would mean not supporting deferred contexts.
81
82 You need to set up all of the resource pools before you can start rendering.
83 If you modify this at runtime, you need to call IggyPlayerFlushAll on all
84 active Iggys (if any) since this call invalidates all resource handles they
85 currently hold.
86
87 Resource pool memory has certain alignment requirements - see the #defines
88 above. If you pass in an unaligned pointer, GDraw will automatically clip off
89 some bytes at the front to make the buffer aligned - in other words, you get
90 somewhat less usable bytes, but it should work fine.
91
92 If any Iggy draw calls are in flight, this call will block waiting for
93 those calls to finish (i.e. for the resource memory to become
94 unused).
95*/
96

108/* Creates a GDraw context for rendering using GXM. You need to pass in a pointer to
109 the shader patcher to use, a pointer to "context memory" which holds a few persistent
110 resources shared by all Iggys (it needs to hold GDRAW_PSP2_CONTEXT_MEM_SIZE bytes
111 and must be mapped to be GPU readable), and a pointer to the notificaiton to use for

113

115 GDraw's fragment shaders. This should match your color surface. Typical choices
116 are:
117 - SCE_GXM_OUTPUT_REGISTER_FORMAT_UCHAR4 (32-bit output register size in color surface)
118 - SCE_GXM_OUTPUT_REGISTER_FORMAT_HALF4 (64-bit output register size in color surface)

134IDOC extern void gdraw_psp2_Begin(SceGxmContext *context, const SceGxmColorSurface *color, const SceGxmDepthStencilSurface *depth,
135 gdraw_psp2_dynamic_buffer *dynamic_buffer);
136/* This sets the SceGxmContext that GDraw writes its commands to. It also specifies
137 the dynamic buffer to use. Any GDraw / Iggy rendering calls outside a
138 Begin / End bracket are an error and will be treated as such. The GXM context
139 and dynamic buffer must be live during the entire Begin / End bracket.
140
141 NOTE: If you are passing in a deferred context, see important notes below!
142
143 GDraw uses the color and depth surfaces for parameter validation. Make sure to
144 pass in the same values you passed in to sceGxmBeginScene. Also, inside a given
145 scene, you may have only *one* $gdraw_psp2_Begin / $gdraw_psp2_End pair.
146
147 GDraw maintains a persistent resource cache shared across all Iggys. Because of this,
148 it is *vital* that all command lists generated from GDraw be executed in the order
149 they were generated, or the resource pools might get corrupted.
150
151 If the dynamic buffer is of insufficient size, GDraw will not be able to allocate dynamic
152 vertex data or upload new texture/vertex buffer data during some frames, resulting in
153 glitching! (When this happens, it will be reported as a warning, so make sure to install
154 a warning callback).
155
156 You should use multiple dynamic buffers (at least double-buffer it); otherwise,
157 GDraw needs to stall every frame to wait for the GPU to process the previous one.
158
159 GDraw starts no scenes of its own; you must call BeginScene before you issue
160 gdraw_psp2_Begin.
161
162 If you are a using a deferred context:
163 --------------------------------------
164
165 It's allowed to pass in a deferred context as "context". You may perform Iggy
166 rendering on a separate thread. However, because there is only a single global
167 resource pool that is modified directly by GDraw, rendering is *not* thread-safe;
168 that is, you may do Iggy rendering on any thread, but only a single thread may
169 be inside a GDraw Begin / End bracket at any given time.
170
171 Furthermore, if you use a deferred context, you must use (and execute!) the
172 resulting command list and end the containing scene *before* you call
173 $gdraw_psp2_Begin again. That is, usage must look like this:
174
175 Main thread: | Other thread:
176 <other work> | gdraw_psp2_Begin(deferred_ctx, ...)
177 | <render Iggys>
178 | gdraw_psp2_End(deferred_ctx, ...)
179 | ...
180 | sceGxmEndCommandList(deferred_ctx, &cmd_list)
181 |
182 sceGxmBeginScene(...) | <other work>
183 sceGxmExecuteCommandList(..., cmd_list) |
184 sceGxmEndScene() |
185
186 The second thread may *not* start another $gdraw_psp2_Begin before the main thread
187 calls sceGxmEndScene().
188
189 This is inconvenient, but unfortunately required by our resource management: every
190 GDraw Begin / End Bracket may end up having to wait for the GPU to finish work done
191 during a previous bracket, and this only works if the corresponding jobs have been
192 submitted to the GPU by the point $gdraw_psp2_Begin is called!
193*/
194
